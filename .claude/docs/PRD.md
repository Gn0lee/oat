# PRD.md

> Product Requirements Document - oat

## TL;DR

- **핵심 가치** - 가족의 돈 흐름을 관리하고, 자산이 자라는 과정을 추적
- **서비스 구조** - 가계부(일상 흐름) + 자산 관리(성장 결과)가 하나로 연결
- **프라이버시** - 공용 지출은 가구 전체 공개, 개인 지출은 본인만 조회
- **현재 상태** - v1.0(주식 관리) 완료, 기능 단위로 순차 확장 중
- **데이터 입력** - 수동 입력 (은행/증권사 연동 불가)
- **타겟** - 자산을 함께 키워나가고 싶은 신혼부부/가족

---

## 1. 제품 개요

### 1.1 제품명
**oat**

'오트밀'처럼 건강하고 매일 챙겨 먹는(확인하는) 습관을 연상시키며, 발음이 가볍고 기억하기 쉽습니다.

### 1.2 한줄 요약
가족이 함께 돈의 흐름을 관리하고, 자산이 자라는 과정을 추적하는 서비스

### 1.3 핵심 철학

> **"돈이 어디에 있고, 어디서 어디로 흐르는가"**

매달의 수입/지출 흐름(과정)이 결국 장기 자산(결과)을 만듭니다.

```
매달의 흐름 (가계부)                 자산의 성장 (자산 관리)

수입 500만                          총 자산 1.25억
 └ 지출 380만                         ├ 투자: 1억 (+12%)
    ├ 식비 85만                       ├ 예적금: 2천만
    ├ 주거 75만                       └ 입출금: 500만
    └ ...
 └ 이체/투자 120만 ──────────────→   매달 여기에 쌓임
```

### 1.4 핵심 가치
| 가치 | 설명 |
|------|------|
| **Daily Flow** | 매일의 수입/지출 흐름을 간편하게 기록하고 파악 |
| **One View** | 가족 구성원의 자산을 하나의 대시보드에서 통합 조회 |
| **Growth Tracking** | 월별 저축률, 자산 추이로 "우리 집 자산이 자라는 과정" 확인 |

## 2. 타겟 사용자

### 2.1 주요 타겟
- 자산을 함께 키워나가고 싶은 **신혼부부**
- 공용 지출과 개인 지출을 체계적으로 관리하고 싶은 가족
- 각자 투자하면서도 전체 자산 현황을 파악하고 싶은 가구
- 엑셀/노션 수기 관리에서 벗어나고 싶은 사람들

### 2.2 사용자 페르소나

**페르소나 A: 김진호 (35세, 남편)**
- 미국 기술주 위주 투자 (애플, 테슬라, 엔비디아)
- 토스증권, 키움증권 사용
- 신용카드 2장, 체크카드 1장 사용
- "이번 달 우리가 얼마 쓴 건지, 카드값이 어디서 빠지는 건지 정리가 안 돼"
- "상품권 할인 사서 장보는데 실제 지출이 얼마인지 헷갈려"

**페르소나 B: 이수진 (33세, 아내)**
- 국내 배당주, ETF 위주 투자 (삼성전자, KODEX 200)
- 삼성증권, 네이버페이 적극 사용
- "공용 지출은 공유하고 싶은데 개인 지출까지 다 보이는 건 불편해"
- "매달 얼마를 모으고 있는지 한눈에 보고 싶어"

### 2.3 확장 타겟 (향후)
- 부모님 자산까지 함께 관리하고 싶은 가족
- 신혼 가계부를 시작하는 예비부부

## 3. 문제 정의

### 3.1 현재 페인포인트

| 문제 | 상세 |
|------|------|
| **지출 흐름 불투명** | 어떤 카드로 얼마 썼고, 그게 어떤 계좌에서 빠지는지 파악 어려움 |
| **공용/개인 지출 구분 어려움** | 가족 공용 지출과 개인 지출이 섞여서 관리 혼란 |
| **전체 자산 파악 어려움** | "우리 지금 다 합쳐서 얼마야?"에 바로 답하기 어려움 |
| **다양한 결제수단 혼란** | 카드, 페이, 상품권 등 결제수단이 다양해서 추적 어려움 |
| **저축률 파악 불가** | 이번 달에 실제로 얼마나 모았는지, 흐름이 좋은 건지 나쁜 건지 판단 어려움 |
| **수기 관리 한계** | 엑셀/노션 관리는 실수 가능성, 업데이트 귀찮음 |

### 3.2 해결 방안

| 문제 | 해결책 |
|------|--------|
| 지출 흐름 불투명 | 계좌/카드별 지출 내역 기록, 카드-계좌 연결 관계 표시 |
| 공용/개인 구분 | 기록 시 공용/개인 선택, RLS로 개인 지출은 본인만 조회 |
| 전체 자산 파악 | 통합 대시보드에서 합산 자산 표시 |
| 다양한 결제수단 | 카드/페이/상품권을 "결제수단"으로 통합 관리 |
| 저축률 파악 불가 | 월별 수입-지출 차액 자동 계산, 저축률 표시. 이체와 투자 매수/매도는 현금흐름 통계에서 제외 |
| 수기 관리 한계 | 간편한 모바일 기록 UI, 정기 지출 자동 등록 |

### 4.1 가계부 기능 (다음 구현 대상)

가계부는 oat의 **일상 사용 핵심 기능**입니다. 매일 수입/지출을 기록하고 월별 흐름을 파악합니다.

#### 핵심 개념: 공용 지출 vs 개인 지출

| 구분 | 공용 지출 | 개인 지출 |
|------|----------|----------|
| 정의 | 가구 공동 생활비 | 본인만의 지출 |
| 예시 | 식비, 주거비, 공과금, 장보기 | 개인 취미, 용돈 사용 |
| 세부 내역 (가게명, 금액, 카테고리) | 가구 구성원 전체 공개 | **기록한 본인만 조회** |
| 월 합계 금액 | 가구 전체에 공개 | **기록한 본인만 조회** |
| 월별 리포트/저축률 | 공용 장부 기준으로 계산 | 내 개인 장부 기준으로 별도 계산 |

> **장부 분리 원칙**: 공용 장부와 내 개인 장부는 같은 구조의 별도 데이터셋입니다. 두 장부의 수입, 지출, 잔액, 저축률은 서로 합산하지 않습니다. 배우자/파트너의 개인 장부는 세부 내역과 합계 모두 내 화면에 노출하지 않습니다.

#### 가계부 기록
| ID | 기능 | 우선순위 | 설명 |
|----|------|----------|------|
| LEDGER-01 | 지출 기록 | 🔴 필수 | 금액, 카테고리, 내용(제목), 결제수단, 공용/개인 구분, 메모 |
| LEDGER-02 | 수입 기록 | 🔴 필수 | 금액, 카테고리, 내용(제목), 입금 계좌, 메모 |
| LEDGER-03 | 이체 기록 | 🟡 권장 | 계좌→계좌 (적금 납입, 투자 이체) 또는 계좌→결제수단 (상품권 구매, 페이 충전) |
| LEDGER-04 | 내역 수정/삭제 | 🔴 필수 | 잘못 입력한 기록 수정 |
| LEDGER-05 | 월별 내역 조회 | 🔴 필수 | 날짜별 그룹핑, 수입/지출 필터 |
| LEDGER-06 | 정기 수입/지출 | 🟡 권장 | 월세, 급여 등 반복 항목 등록 |
| LEDGER-07 | 카테고리 관리 | 🔴 필수 | 기본 제공 + 가구별 추가/수정/삭제 |
| LEDGER-08 | Inline Reference Creation | 🔴 필수 | 기록 입력 중 필요한 카테고리, 결제수단, 입금 계좌가 없으면 현재 입력 흐름 안에서 최소 정보로 생성하고 즉시 선택 |

#### 기록 입력 중 참조 데이터 생성

기록 입력 흐름에서 카테고리, 결제수단, 입금 계좌가 없으면 별도 관리 화면으로 이동하지 않고 현재 입력 흐름 안에서 생성할 수 있어야 합니다. 생성 후에는 방금 만든 항목이 현재 기록 필드에 즉시 선택되고, 사용자가 입력 중이던 기록 내용은 유지됩니다.

Inline 생성은 기록을 계속하기 위한 최소 정보만 받습니다. 카테고리 아이콘, 계좌 잔액, 계좌번호 뒤 4자리, 결제수단 카드번호 뒤 4자리, 결제일, 보조잔액, 메모 같은 상세 설정은 각 관리 화면에서 수정합니다.

#### 카테고리 관리 방식

기본 카테고리를 제공하되, 가구별로 자유롭게 커스텀 가능합니다.
- **기본 카테고리**: 가입 시 자동 생성 (아래 목록). `is_system = true`로 표시
- **추가**: 가구에 맞는 카테고리 자유 추가 (예: "반려동물", "육아")
- **수정**: 이름, 아이콘 변경 가능 — **단, `is_system = true` 기본 카테고리는 수정/삭제 불가** (DB RLS로 강제)
- **삭제**: `is_system = false`인 커스텀 카테고리만 삭제 가능. 해당 카테고리에 기록이 있으면 `category_id = null`로 변경 후 삭제 (참조 정합성 유지)
- **순서 변경**: `display_order` 컬럼으로 관리. 자주 쓰는 카테고리를 위로 배치 가능
- **아이콘**: Lucide 아이콘명 문자열로 저장 (예: `'utensils'`). UI에서 큐레이션된 목록 중 선택. null이면 기본 아이콘 표시

#### 이체 기록 설계

이체는 내 돈 위치 사이의 이동입니다. 수입/지출 통계에는 포함하지 않고, 각 계좌/결제수단 상세에서는 선택한 대상 기준으로 들어온 돈은 `+`, 나간 돈은 `-`로 표시합니다.

| 이체 유형 | from | to | 예시 |
|----------|------|----|------|
| 계좌→계좌 | 출금 계좌 | 입금 계좌 | 적금 납입, CMA 이동 |
| 계좌→결제수단 | 출금 계좌 | 페이/상품권 | 카카오페이 충전, 상품권 구매 |
| 결제수단→계좌 | 페이/상품권/현금 | 입금 계좌 | 페이머니 환급, 현금 입금 |

> 이체 기록은 가구 전체에 공개(`is_shared = true`)로 기록하는 것을 권장합니다. 타인에게 보낸 돈은 내 돈 위치 사이의 이동이 아니므로 MVP에서는 지출로 기록합니다.

#### 잔액 관리 원칙

계좌와 일부 결제수단은 현재 잔액과 잔액 변동 내역을 관리합니다. 단, 잔액 관리 대상과 총자산 합산 대상은 다릅니다.

- 계좌 잔액은 총자산에 포함합니다.
- 투자 계좌의 `balance`는 예수금입니다. 투자 계좌 총자산 화면에서는 예수금과 보유 주식 평가액을 합산해 보여줄 수 있습니다.
- 선불지갑, 상품권, 현금은 잔액과 거래내역을 관리하지만 MVP의 총자산에는 포함하지 않습니다.
- 신용카드와 체크카드는 자체 잔액을 갖지 않습니다.
- 저장된 `balance` 값을 현재 잔액으로 신뢰합니다. 기록 내역은 잔액 변화의 근거이며 MVP에서 잔액 재계산의 원천은 아닙니다.

잔액에 영향을 주는 기록은 저장 시점에 잔액을 함께 갱신합니다.

- 수입: 입금 계좌 또는 잔액 있는 결제수단 증가
- 지출: 출금 계좌 또는 잔액 있는 결제수단 감소
- 이체: 출발 대상 감소, 도착 대상 증가
- 체크카드 지출: 체크카드 자체 잔액은 없지만, 연결 계좌가 있으면 연결 계좌 잔액 감소
- 체크카드에 연결 계좌가 없으면 지출 저장은 허용하되 계좌 잔액에는 반영하지 않습니다. 사용자는 연결 계좌가 없어 잔액 반영이 되지 않는다는 안내를 받아야 합니다.
- 체크카드 연결 계좌를 나중에 설정하거나 변경해도 과거 지출을 자동 소급 반영하지 않습니다. 새 설정은 앞으로 생성되는 지출에만 적용하고, 과거 차이는 `실제 잔액 맞추기`로 처리합니다.
- 신용카드 지출: 구매 시점에 지출로 기록하고 계좌 잔액은 감소시키지 않음
- 주식 매수/매도: 연결된 투자 계좌의 예수금 감소/증가
- 잔액 조정: 사용자가 입력한 실제 잔액으로 대상 잔액을 맞춤
- 저장 후 관리 잔액이 음수가 될 수 있으면 제출 전 확인 Dialog를 보여줍니다. 사용자가 확인하면 저장을 허용하고, 상세 화면에서 실제 잔액 맞추기를 유도합니다.

#### 잔액 조정

잔액 조정은 실제 수입/지출이 아니라 앱 잔액과 실제 잔액의 차이를 맞추기 위한 기록입니다.

- 액션 이름: `실제 잔액 맞추기`
- 기본 기록 제목: `잔액 맞춤`
- 기록 타입 표시: `잔액 조정`
- 수입/지출 통계와 저축률에는 포함하지 않습니다.
- 계좌/결제수단 상세 거래내역에는 표시합니다.
- 일반 일별 가계부 목록에는 기본 노출하지 않습니다.
- 해당 계좌/결제수단 소유자만 생성할 수 있습니다.

#### 카드 결제대금

MVP에서는 신용카드 명세/청구/결제대금 납부 생애주기를 모델링하지 않습니다.

- 신용카드 구매는 구매 시점에 지출로 기록합니다.
- 이후 은행 계좌에서 카드값이 출금되는 이벤트는 별도로 기록하지 않습니다.
- 카드값 출금 때문에 계좌 실제 잔액과 앱 잔액이 다르면 `실제 잔액 맞추기`로 맞춥니다.
- 향후 은행 계좌 상세에서 카드값 출금 내역이 필요해지면 `card_payment` 이벤트를 추가합니다. 이 이벤트는 계좌 잔액과 카드/계좌 상세 내역에는 반영하지만, 이미 구매 시점에 잡힌 지출 통계에는 다시 포함하지 않습니다.

#### 개인 장부 노출 방식

`is_shared = false`인 항목은 DB RLS로 기록한 본인(`owner_id = auth.uid()`)만 조회 가능합니다.
파트너에게 개인 장부의 세부 내역이나 월 합계를 노출하지 않습니다.

#### 용돈 기록 방식

용돈 지급은 공용 장부의 지출로 기록하고, 받은 용돈은 개인 장부의 수입으로 기록할 수 있습니다. 이후 용돈 사용은 개인 장부의 지출입니다. 공용 장부와 개인 장부를 합산하지 않으므로 용돈 지급과 용돈 사용이 이중 집계되지 않습니다.

### 4.2 계좌/결제수단 관리 (다음 구현 대상)

**자산**과 **결제수단**은 명확히 구분됩니다.
- **자산 (계좌)**: 돈이 있는 곳 → **총 자산에 포함**
- **결제수단**: 돈이 나가는 경로 → **총 자산에 미포함**, 지출 흐름 추적용

```
[은행 계좌 (자산)] ← 실제 돈이 있는 곳
    ├── 신용카드 (결제수단) → 매달 결제일에 계좌에서 차감
    ├── 체크카드 (결제수단) → 즉시 계좌에서 차감
    └── 네이버페이 (결제수단) → 충전 시 계좌에서 차감
```

| ID | 기능 | 우선순위 | 설명 |
|----|------|----------|------|
| ACCT-01 | 계좌(자산) 등록 | 🔴 필수 | 은행 계좌, 증권 계좌 등 |
| ACCT-02 | 결제수단 등록 | 🔴 필수 | 카드, 페이, 상품권 등 — **UI는 Epic #3 (가계부) 에서 구현** |
| ACCT-03 | 결제수단-계좌 연결 | 🟡 권장 | 카드가 어떤 계좌에서 결제되는지 — **Epic #3에서 구현** |
| ACCT-04 | 잔액 기록 | 🟡 권장 | 계좌별 현재 잔액 수동 입력/갱신 |
| ACCT-05 | 계좌 목록 조회 | 🔴 필수 | 계좌(자산) 전체 목록 (`/assets/accounts`) |

Inline Reference Creation에서는 가계부 계좌를 `bank/checking`, 주식 거래 계좌를 `investment/stock`으로 기본 생성합니다. 지출 money source에서는 신용카드, 체크카드, 선불페이, 상품권, 현금, 계좌 중 생성 대상을 먼저 고르고, 결제수단 생성 시 연결 계좌는 선택사항입니다.

> **결제수단 UI 위치 변경**: 결제수단은 자산이 아닌 지출 흐름 추적 도구이므로, 관리 화면은 가계부(`/ledger/payment-methods`) 영역에 위치합니다. DB/API는 Epic #2에서 완료, UI는 Epic #3에서 구현합니다.

#### 계좌 = 자산 (총 자산에 포함)

실제 돈이 보관되는 곳. 잔액이 총 자산 합산에 포함됩니다.

| 유형 | 설명 | 예시 |
|------|------|------|
| 입출금 | 월급통장, 생활비통장 | KB국민 월급통장 |
| 예금 | 정기예금 | 토스뱅크 예금 |
| 적금 | 정기적금, 자유적금 | 카카오뱅크 적금 |
| 증권 | 주식/ETF 계좌 (기존 accounts와 통합) | 키움증권 |
| CMA/연금 | CMA, IRP, ISA | 미래에셋 IRP |

#### 결제수단 (총 자산에 미포함, 지출 흐름 추적용)

지출 시 "어떻게 결제했는지"를 추적하기 위한 수단. 자산이 아닙니다.

| 유형 | 설명 | 연결 정보 |
|------|------|----------|
| 신용카드 | 후불 결제 | 결제 계좌, 결제일 |
| 체크카드 | 즉시 출금 | 출금 계좌 |
| 선불/페이 | 충전식 (네이버페이, 카카오페이) | 충전 원천 계좌 |
| 상품권 | 할인 구매 가능 | 구매 시 출금 계좌 |
| 현금 | 실물 현금 | - |

### 4.3 가계부 분석 대시보드 (/ledger/analysis)

가계부 홈에서 진입. 현재 월 및 최근 추이를 분석하는 통계 대시보드. **가계부 전용 독립 페이지**로 구성 (주식 분석 `/assets/stock/analysis`와 분리).

#### 데이터 범위 (scope)

| scope | 설명 |
|-------|------|
| `shared` | 공용 장부 항목만 (가구 전체 동일한 뷰) |
| `personal` | 로그인한 본인의 개인 장부 항목만 |

> `shared`와 `personal`은 별도 데이터셋이며 서로 합산하지 않습니다. 파트너의 개인 장부는 세부 내역과 합계 모두 노출하지 않습니다.

#### 화면 섹션 및 API

| ID | 기능 | 우선순위 | scope 선택 | API |
|----|------|----------|:---:|-----|
| FLOW-01 | 공용/내 개인 월별 요약 | 🔴 필수 | ✅ | `GET /api/ledger/stats/summary` |
| FLOW-02 | 멤버별 공용 지출 | 🔴 필수 | - | `GET /api/ledger/stats/by-member` |
| FLOW-03 | 카테고리별 지출 비율 | 🔴 필수 | ✅ | `GET /api/ledger/stats/by-category` |
| FLOW-04 | 결제수단별 지출 | 🟡 권장 | ✅ | `GET /api/ledger/stats/by-payment-method` |
| FLOW-05 | 저축률 표시 | 🟡 권장 | - | summary API에 포함 |
| FLOW-06 | 월별 추이 (최근 N개월) | 🟡 권장 | - | `GET /api/ledger/stats/trend` |
| FLOW-07 | 일별 지출 분포 | 🟡 권장 | ✅ | `GET /api/ledger/stats/daily` |

#### API 파라미터 요약

| API | 주요 파라미터 | 응답 주요 필드 |
|-----|-------------|--------------|
| summary | `year`, `month` | `shared{totalIncome,totalExpense,balance,savingsRate}`, `personal{totalIncome,totalExpense,balance,savingsRate}` |
| by-member | `year`, `month` | `members[]{sharedExpense}` |
| by-category | `year`, `month`, `type`, `scope` | `items[]{categoryName, amount, percentage}` |
| by-payment-method | `year`, `month`, `scope` | `items[]{paymentMethodName, amount, percentage}` |
| trend | `months` (기본 6, 최대 12) | `items[]{year, month, totalExpense, savingsRate}` |
| daily | `year`, `month`, `scope` | `items[]{date, totalIncome, totalExpense}` (캘린더 히트맵용) |

#### 분석 드릴다운 요구사항

분석 화면은 요약 수치를 보여주는 데서 끝나지 않고, 사용자가 `N건`, 금액, 보유 종목 수를 보고 궁금해할 원본 항목을 현재 분석 맥락 안에서 확인할 수 있어야 합니다.

1차 드릴다운 대상:

| 화면 | UI | 동작 |
|------|----|------|
| `/ledger/analysis/by-category` | 상세 내역 row | 해당 카테고리 원본 가계부 기록 drawer |
| `/ledger/analysis/by-payment-method` | 상세 내역 row | 해당 결제수단 원본 가계부 기록 drawer |
| `/ledger/analysis/trend` | 수입/지출 막대 | 해당 월 기록 페이지 이동 확인 dialog |
| `/ledger/analysis/trend` | 월별 상세 수입/지출 영역 | 해당 월 기록 페이지 이동 확인 dialog |
| `/ledger/analysis/daily` | 일별 지출 막대 | 해당 날짜 지출 기록 drawer |
| `/assets/stock/analysis` | 종목별 비중 row | 해당 종목의 보유 row와 최근 거래 drawer |
| `/assets/stock/analysis` | 계좌별 비중 row | 해당 계좌의 보유 종목과 최근 거래 drawer |
| `/assets/stock/analysis` | 종목별 계좌 분포 row | 해당 종목+계좌의 보유 정보와 관련 거래 drawer |
| `/assets/stock/analysis` | 수익/손실 TOP row | 종목 drawer 재사용 |

1차 제외 대상:

- 카테고리/결제수단 도넛 조각과 범례: 상세 내역 row로 커버합니다.
- 카테고리 전월 대비 차트: 차트 의미 개선이 먼저 필요합니다.
- 구성원별 지출 페이지: 페이지 자체의 분석 의미를 먼저 개선합니다.
- 월 요약, 평균, 저축률: 원본 목록 의미가 약하거나 drawer에 담기엔 범위가 넓습니다.
- 주식 도넛 조각: row 클릭으로 커버하며 모바일 터치 타겟이 애매합니다.
- 시장별/통화별 비중: 데이터 범위가 넓어 1차 drawer에 부적합합니다.
- 시장 동향: 내 보유/거래 원본과 직접 연결되지 않습니다.

### 6.2.1 기록 입력 UX 원칙

가계부와 주식 거래처럼 여러 기록을 한 번에 만들 수 있는 화면은 단순 카드 누적 폼보다 **Entry Composer** 패턴을 우선합니다.

- 일반 진입은 `full` route를 사용하고, 사용자가 공개범위/유형/날짜/계좌 같은 기본값을 정한 뒤 여러 행을 입력합니다.
- 캘린더처럼 날짜 맥락이 있는 진입은 `daily` route를 사용하고, 선택한 날짜를 기본값으로 바로 입력을 시작합니다.
- 주식 계좌 상세처럼 계좌 맥락이 있는 진입은 `account` route를 사용하고, 선택한 계좌를 기본값으로 바로 입력을 시작합니다.
- 각 행은 기본값을 상속하되, 필요한 경우 행 단위 상세 편집에서 유형/날짜/계좌/결제수단 등을 override할 수 있습니다.
- 모바일에서는 여러 상세 카드가 동시에 펼쳐지지 않도록 압축 행 목록과 단일 행 편집 화면을 우선합니다.

계좌, 결제수단, 카테고리처럼 기록 중 필요한 참조 데이터는 장기적으로 **Inline Reference Creation**을 지원합니다. 사용자가 기록 입력 화면을 떠나 별도 관리 화면에서 생성한 뒤 돌아오는 흐름은 피합니다.

## 7. 비기능 요구사항

### 7.1 성능

| 항목 | 목표 |
|------|------|
| 페이지 로드 | 3초 이내 (FCP) |
| API 응답 | 500ms 이내 |
| 가계부 기록 | 탭 3번 이내로 기록 완료 |

### 7.2 보안 및 프라이버시

| 항목 | 요구사항 |
|------|----------|
| 인증 | 이메일/비밀번호 |
| 데이터 접근 | RLS로 가구 단위 데이터 격리 |
| **개인 지출 세부 내역** | RLS로 `owner_id = auth.uid()`인 경우만 조회 가능 (메모, 카테고리, 개별 금액). 가구 `owner` 역할과 무관 |
| **개인 장부 합계** | 기록한 본인만 조회 가능. 파트너에게 합계도 노출하지 않음 |
| **공용 지출** | `is_shared = true`. 같은 가구 구성원 전체 조회 가능 |
| **이체 기록** | `is_shared = true` 권장. 가구 현금 흐름 계산에 포함 |
| 통신 | HTTPS 필수 |

### 7.3 접근성

| 항목 | 요구사항 |
|------|----------|
| 반응형 | 모바일 우선 (PWA 기반) |
| PWA | 앱 설치, 오프라인 기본 화면 |
| 서비스 헤더 | 모바일은 앱형 헤더와 하단 탭을 사용하고, PC는 사이드바와 콘텐츠 상단 breadcrumb로 현재 위치를 안내 |
