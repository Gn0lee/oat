# DESIGN.md

> UI/UX 원칙 및 디자인 가이드

## TL;DR

- **토스식 정보 위계** - 큰 숫자, 쉬운 금융 언어, 빠른 스캔, 명확한 다음 행동
- **데이터 중심** - 핵심 숫자가 가장 먼저 눈에 들어와야 함
- **반응형** - 모바일(하단 탭 + 앱형 헤더), PC(사이드바 + breadcrumb top bar)
- **컬렉션 UI 우선** - 사용자 화면에서 테이블보다 리스트, 캘린더, 타임라인, 선택-상세 구조를 우선 검토
- **컬러** - 인디고 블루 기반, 상승 빨강/하락 파랑 (한국식)
- **다크모드** - MVP 제외, 2단계에서 추가

## 1. 디자인 원칙

### 1.1 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **데이터 중심** | 숫자와 핵심 정보가 가장 먼저 눈에 들어와야 함 |
| **단순함** | 한 화면에 하나의 목적, 불필요한 요소 제거 |
| **직관적 흐름** | 설명 없이도 다음 행동을 알 수 있어야 함 |
| **신뢰감** | 금융 서비스답게 안정적이고 깔끔한 인상 |

### 1.2 토스에서 배울 점

oat가 참고하는 토스 스타일은 둥근 카드 외형이 아니라 정보 위계와 쉬운 금융 언어입니다.

가져올 것:

- 큰 숫자와 작은 레이블로 핵심 상태를 먼저 보여줍니다.
- `₩` 기호보다 `원` 중심의 친숙한 금액 표현을 우선 검토합니다.
- 사용자가 빠르게 훑을 수 있는 리스트와 행 기반 구조를 기본값으로 둡니다.
- 큰 터치 영역과 명확한 다음 행동을 제공합니다.
- 중요한 상태와 변화만 색으로 강조합니다.

가져오지 않을 것:

- 모든 영역을 둥근 카드로 감싸는 화면 구성
- 과한 여백 때문에 정보량이 줄어드는 화면
- 마케팅성 일러스트나 감성 문구 중심 화면
- PC에서 모바일 카드를 그대로 여러 장 배치하는 방식

## 2. 레이아웃

### 2.1 서비스 레이아웃 구조

모바일과 PC는 같은 헤더를 늘려 쓰지 않습니다. 모바일은 앱처럼 화면 성격에 따라 상단 affordance를 바꾸고, PC는 좌측 사이드바와 콘텐츠 상단 breadcrumb로 위치를 안내합니다.

**모바일 Top-Level Screen**
```
┌─────────────────────────────────────────┐
│  Service Header (oat 로고)               │
├─────────────────────────────────────────┤
│                                         │
│              Main Content               │
│        (큰 페이지 제목 반복 없음)          │
│                                         │
├─────────────────────────────────────────┤
│  Bottom Navigation                      │
└─────────────────────────────────────────┘
```

**모바일 Child/Task Screen**
```
┌─────────────────────────────────────────┐
│  ←  화면 제목                    (X)     │
├─────────────────────────────────────────┤
│                                         │
│              Main Content               │
│        (큰 페이지 제목 반복 없음)          │
│                                         │
└─────────────────────────────────────────┘
```

**PC**
```
┌──────────────┬──────────────────────────┐
│ oat 로고      │  Breadcrumb               │
│              ├──────────────────────────┤
│  Sidebar     │                          │
│  Navigation  │       Main Content        │
│              │  (큰 페이지 제목 반복 없음) │
└─────────────────────────────────────────┘
```

### 2.3 네비게이션과 서비스 헤더

**모바일**: 하단 탭 네비게이션 + 화면 성격별 Service Header
```
[홈]  [가계부]  [자산]  [설정]
```

- Top-level 화면(`/home`, `/ledger`, `/assets`, `/settings`)은 로고형 Service Header를 사용하고 본문 큰 제목을 반복하지 않습니다.
- Top-level hub는 현재 상태 요약과 핵심 행동을 함께 보여주되, 모든 기능을 같은 무게로 나열하지 않습니다.
- 홈 hub의 핵심 행동은 가계부와 자산입니다. 설정은 관리 행동이고, 분석은 통합 분석 hub가 생기기 전까지 가계부/자산에서 파생되는 탐색으로 봅니다.
- Child 화면은 `뒤로가기 + 화면 제목` 앱 바를 사용합니다.
- Task 화면은 필요 시 우상단 `X`를 제공합니다. `X`는 이전 화면이 아니라 작업 흐름을 닫고 안정적인 상위 목적지로 나가는 Close Action입니다.

**PC**: 사이드 네비게이션 + 콘텐츠 top bar breadcrumb
```
┌──────────────┐
│ oat 로고      │
├──────────────┤
│ 홈           │ → /home
│ 가계부        │ → /ledger
│ 자산         │ → /assets
│ 설정         │ → /settings
└──────────────┘
```

- 콘텐츠 영역 상단의 흰색 bar 좌측에 breadcrumb를 둡니다.
- PC 본문에서도 큰 페이지 제목을 반복하지 않습니다.
- 사이드바는 MVP에서 1차 메뉴만 유지할 수 있습니다. 계층형 사이드바는 후속 개선 방향입니다.
- 사이드바 상단은 일단 oat 로고만 둡니다. 가구/워크스페이스 identity는 별도 제품 개념으로 설계될 때 추가합니다.

**자산 페이지 내부**: 유형별 탭 또는 카드
```
/assets
├── 전체 보유 현황 → /assets/total/holdings
├── 주식/ETF      → /assets/stock/...
├── 현금 (준비중)  → /assets/cash/...
├── 부동산 (준비중) → /assets/real-estate/...
└── 기타 (준비중)  → /assets/other/...
```

## 3. 컬러

### 3.1 컬러 선정 원칙

| 원칙 | 설명 |
|------|------|
| 신뢰감 | 금융 서비스에 어울리는 안정적인 톤 |
| 현대적 | 올드하거나 투박하지 않은 느낌 |
| 절제 | 색상은 최소화, 강조점에만 사용 |

**추천 계열**
- 블루/인디고 계열: 신뢰, 안정
- 그린/틸 계열: 성장, 금융
- 퍼플 계열: 현대적, 프리미엄

### 3.2 컬러 구조

```css
:root {
  /* Primary - 인디고 블루 (신뢰감, 현대적) */
  --primary: #4F46E5;        /* Tailwind indigo-600 */
  --primary-dark: #4338CA;   /* indigo-700 */
  --primary-light: #EEF2FF;  /* indigo-50 */
  
  /* Neutral - 텍스트, 배경 */
  --gray-900: #191F28;       /* 메인 텍스트 */
  --gray-700: #4E5968;       /* 서브 텍스트 */
  --gray-500: #8B95A1;       /* 비활성 텍스트 */
  --gray-300: #D1D6DB;       /* 보더 */
  --gray-100: #F2F4F6;       /* 배경 */
  --white: #FFFFFF;
  
  /* Semantic - 의미 기반 */
  --positive: #F04452;       /* 상승 (빨강) */
  --negative: #3182F6;       /* 하락 (파랑) */
  --warning: #FF9F00;
  --success: #03B26C;
}
```

### 3.3 컬러 사용 규칙

| 용도 | 컬러 |
|------|------|
| 메인 텍스트 | `gray-900` |
| 보조 텍스트 | `gray-700` |
| 비활성/힌트 | `gray-500` |
| 주요 액션 버튼 | `primary` |
| 수익 상승 | `positive` (빨강) |
| 수익 하락 | `negative` (파랑) |
| 배경 | `gray-100` 또는 `white` |
| 카드 배경 | `white` |

### 3.4 다크모드 (2단계)

MVP에서는 라이트모드만 지원. 2단계에서 다크모드 추가 예정.

```css
/* 2단계에서 추가 */
[data-theme="dark"] {
  --gray-900: #FFFFFF;
  --gray-100: #191F28;
  /* ... */
}
```

## 4. 타이포그래피

### 4.1 폰트

```css
font-family: 'Pretendard', -apple-system, BlinkMacSystemFont, sans-serif;
```

### 4.2 크기 스케일

| 용도 | 크기 | 클래스 | 예시 |

```

```

### 4.4 금액과 긴 텍스트 표시

사용자 인터페이스에서 금액(돈) 및 긴 텍스트를 처리할 때는 다음의 정형화된 정책을 따릅니다.

#### 1) 통화 및 포맷팅 정책
- **KRW 기본 금액**: `₩10,000` 형태의 원화 기호를 피하고 한글 접미사 `원`을 사용하여 `10,000원`으로 렌더링합니다.
- **KRW 축약 금액**: 목록 행 등 공간이 좁은 곳에서는 원화 기호 없이 한글 단위를 사용하여 `41.4만원`, `1.2억원`으로 표시합니다.
- **USD 금액**: 달러 기호가 익숙하므로 `$123.45`, `$1.2K` 처럼 기존의 달러 기호와 영어 축약어 형식을 유지합니다.
- **포맷터와 부호**: 통화 포맷터 자체는 임의로 부호(sign)를 소유하지 않습니다. 부호 결정은 화면의 도메인 성격에 따라 개별적으로 수행됩니다.

#### 2) 화면별 부호 정책 (Sign Matrix)
- **절대적 값**: 총자산, 예수금, 평가금액, 현재가, 폼 입력값 등 단순 상태 스냅샷이나 입력 필드에는 긍정 부호 `+`를 붙이지 않습니다. (단, 음수일 때는 `-`가 자연스럽게 포맷터에 의해 렌더링됩니다.)
- **조밀한 주식 목록**: 주식 보유 목록 등 이미 색상과 배지 등으로 Gain/Loss 정보가 제공되는 조밀한 테이블 행에서는 `+` 부호를 붙이지 않고 숫자만 표시하여 시각적 혼선을 줄입니다 (단, negative `-`는 손실 방향 파악을 위해 유지합니다).
- **가계부 cash-flow 목록**: 수입은 `+`, 지출 및 비지출 출금은 `-`, 내부이체는 부호를 표시하지 않습니다.
- **변화량/Deltas**: 평가손익, 수익률, 계좌 잔액 변화량 등의 지표 영역에서는 증가 시 `+`, 감소 시 `-` 부호를 명시하여 변화를 한눈에 알아볼 수 있도록 합니다.

#### 3) 금액 노출 및 레이아웃 정책
- **목록 행(Rows)**: 목록 행에는 축약 금액을 기본으로 보여주고, 마우스 오버 시 `title` 툴팁 및 스크린 리더용 `aria-label`을 제공하여 전체 상세 금액(`12,345,678원`)을 확인할 수 있도록 지원합니다.
- **상세/지표(Details/Metrics)**: 상세 페이지, 확인 화면, 요약 지표 영역에서는 축약되지 않은 전체 금액을 기본으로 표시합니다.
- **지표 그리드(Metrics Grid)**: 모바일 2열 격자 등 좁은 영역의 지표는 금액이 길어질 때 텍스트가 서로 겹치거나 잘리지 않도록 `min-w-0`, wrapping(`[overflow-wrap:anywhere]`), 모바일용 폰트 사이즈 조정(기본 모바일 `text-base` 지정 및 `sm:text-lg` 대응)을 필수로 적용합니다.
- **오른쪽 금액 스택**: 리스트 행의 우측 trailing 영역에 여러 줄의 금액을 렌더링할 때는 주식명이나 가계부 제목 영역을 과도하게 침범하지 않도록 trailing 영역의 너비를 명시적으로 제한합니다(예: `w-[7.5rem] max-w-[40vw] text-right`).

#### 4) 텍스트 생략 및 줄바꿈 정책
- **주요 라벨 (가계부 제목, 주식명 등)**: 너무 길 경우 뒷부분이 생략되어 의미가 불분명해지는 것을 막기 위해 `line-clamp-2` 및 `[word-break:break-all]`, `whitespace-normal`을 적용하여 최대 2줄까지 줄바꿈 렌더링을 보장합니다.
- **보조 메타데이터 (소유자, 계좌명, 카테고리 등)**: 1줄 생략(`truncate`)을 적용하며, 이를 포함하는 flex 컨테이너에 `min-w-0`을 주어 레이아웃이 깨지지 않게 방지합니다.
- **가계부 메모**: 요약 목록 행에서는 메모를 생략하며, 상세 조회 화면에서만 줄바꿈(`whitespace-pre-wrap`)하여 전체 내용을 렌더링합니다.

#### 5) 폼(Forms) 입력 유닛 가이드라인
- **Prefix 제거**: 활성 입력 필드 내부에 장식용 `₩` 접두사를 삽입하지 않습니다.
- **레이블 명시**: `금액 (원) *`, `단가 (원) *`, `단가 ($) *`와 같이 입력 필드 라벨을 통해 입력 단위를 사용자에게 명확히 전달하며, 입력 필드 하단의 미리보기 영역에는 포맷팅된 형태(`10,000원`)로 즉각 피드백을 제공합니다.

## 5. 컴포넌트 가이드

### 5.0 돈 위치와 이체 UX

사용자가 수입/지출/이체를 회계 용어로 판단하게 만들지 않고, 돈이 어디에 있고 어디로 움직였는지를 기준으로 입력하게 합니다.

- 이체 입력 라벨은 `출발지/도착지`보다 `어디에서/어디로`를 우선 사용합니다.
- 이체 예시는 `카카오페이 충전`, `증권 계좌 입금`, `적금 납입`, `페이머니 환급`처럼 내 돈 위치 사이의 이동을 보여줍니다.
- 안내 문구는 `타인에게 보낸 돈은 이체가 아니라 지출로 기록해주세요.`를 유지합니다.
- 일반 일별 가계부 리스트에서는 이체를 `토스뱅크 → 카카오페이머니`처럼 중립적으로 한 줄 표시합니다.
- 특정 계좌/결제수단 상세에서는 선택한 대상 기준으로 들어온 돈은 `+`, 나간 돈은 `-`로 표시합니다.
- 이체 아이콘은 카테고리 fallback 아이콘이 아니라 `ArrowLeftRight`를 사용합니다.

잔액 조정 UX:

- 액션 버튼: `실제 잔액 맞추기`
- 기본 제목: `잔액 맞춤`
- 타입 배지: `잔액 조정`
- 설명: `앱 잔액과 실제 잔액의 차이를 맞춘 기록이에요. 수입/지출 통계에는 포함되지 않아요.`

계좌/결제수단 상세 UX:

- 계좌 상세는 현재 잔액과 잔액 변화 타임라인을 함께 보여줍니다.
- 투자 계좌 상세는 예수금과 보유 주식 평가액을 구분하고, 필요하면 합산 총액을 별도 표시합니다.
- 선불지갑/상품권/현금 상세는 잔액을 보여주되 총자산에는 포함되지 않는 보조 잔액임을 안내합니다.
- 신용카드/체크카드 상세는 지출 내역 중심으로 보여주고 자체 잔액 영역을 두지 않습니다.
- 체크카드에 연결 계좌가 없을 때 지출을 기록하면 `연결 계좌가 없어 계좌 잔액에는 반영되지 않아요.`처럼 강한 안내를 보여줍니다.
- 체크카드 연결 계좌 설정 화면에서는 `앞으로 기록하는 체크카드 지출부터 연결 계좌 잔액에 반영돼요.`처럼 과거 기록 자동 소급이 없음을 알려줍니다.
- 기록 저장 후 관리 잔액이 음수가 될 것으로 예상되면 제출 전 확인 Dialog를 보여줍니다. 예: `저장하면 토스뱅크 잔액이 -12,000원이 됩니다. 그래도 저장할까요?`
- 음수 잔액 확인 Dialog의 보조 문구는 `실제 잔액과 다르면 나중에 실제 잔액 맞추기로 조정할 수 있어요.`를 사용합니다.
- 확인 버튼은 `그래도 저장`, 취소 버튼은 `다시 확인`을 사용합니다.

### 5.1 화면 단위와 카드

화면을 나누는 기본 단위는 카드가 아니라 섹션과 리스트 row입니다. 카드는 모든 영역을 감싸는 장식 프레임이 아니라, 독립적으로 강조하거나 선택해야 하는 정보 덩어리에만 사용합니다.

기본 화면 구성:

- 페이지는 `요약 섹션 → 주요 작업 → 기록/자산 리스트` 흐름을 우선합니다.
- 섹션은 제목, 여백, divider, 배경 차이로 구분합니다.
- 반복 데이터는 카드보다 row를 기본값으로 둡니다.
- row는 `대표 텍스트 + 보조 정보 + 금액/상태 + 액션 affordance` 구조를 우선합니다.

카드를 사용할 수 있는 경우:

- 총 자산, 월 지출, 수익률처럼 독립적으로 강조해야 하는 요약 지표
- 자산 유형, 분석 유형처럼 선택 가능한 큰 진입점
- 빈 상태, 권한/설정 안내처럼 화면의 주 목적을 보조하는 단일 메시지
- Dialog/Drawer 안에서 다른 콘텐츠와 명확히 분리해야 하는 확인 영역

카드를 피해야 하는 경우:

- 페이지 안의 일반 섹션 구분
- 날짜별 기록, 보유 종목, 계좌, 결제수단, 알림처럼 반복되는 목록
- PC에서 빈 공간을 채우기 위해 모바일 카드를 여러 장 늘어놓는 구성
- 테이블 컬럼을 그대로 옮긴 정보 나열

### 5.2 버튼

```tsx
// Primary
<Button className="w-full bg-primary text-white rounded-xl py-3">
  자산 추가하기
</Button>

// Secondary
<Button variant="outline" className="rounded-xl">

</Button>
```

| 타입 | 용도 |
|------|------|
| Primary | 주요 액션 (저장, 추가, 확인) |
| Secondary/Outline | 보조 액션 (취소, 뒤로) |
| Ghost | 덜 중요한 액션 |

### 5.5 컬렉션 UI

사용자가 금융 기록이나 자산을 훑어보는 화면은 테이블을 기본값으로 두지 않습니다. PC와 모바일 모두 데이터 성격에 맞는 컬렉션 UI를 우선 검토합니다.

| 데이터 성격 | 권장 UI | 예시 |
|------------|---------|------|
| 날짜가 핵심인 기록 | 캘린더, 날짜별 목록, 타임라인 | 주식 거래 내역, 가계부 기록 |
| 비교가 필요한 보유 항목 | 넓은 리스트, 요약 metric, 선택-상세 | 주식 보유 종목 |
| 설정/관리 항목 | compact list, grouped list | 종목 설정, 계좌, 결제수단 |
| 사람/초대 상태 | 사람 중심 리스트, 상태별 섹션 | 가구 구성원, 대기 중 초대 |
| 월별/기간별 요약 | metric list, timeline, 현재 기간 강조 | 월별 상세 |

원칙:
- 테이블 컬럼을 그대로 카드나 row에 옮기지 않습니다.
- 각 항목은 대표값 1개, 보조 정보 2-4개, 필요한 액션으로 재구성합니다.
- 정렬/필터는 테이블 헤더 클릭이 아니라 상단 control로 제공합니다.
- PC의 넓은 화면은 더 많은 컬럼보다 그룹핑, 비교, 선택-상세 영역에 사용합니다.
- 카드는 화면 전체를 감싸는 기본 프레임이 아니라 제한된 강조/선택 단위로만 사용합니다.
- 내부 table primitive나 TanStack Table 의존성 제거는 별도 정리 대상입니다.

화면별 기본 방향:

| 화면 | 권장 구성 |
|------|----------|
| 주식 거래 내역 | 월 캘린더 또는 날짜 summary, 선택 날짜 거래 목록, 거래 상세/수정 흐름 |
| 주식 보유 현황 | 종목 리스트, 투자금/수량/평균가 중심, 소유자/계좌/시장 badge |
| 종목 설정 | 종목명/티커 중심 compact list, 시장/위험도 badge, 편집 액션 |
| 계좌 | 은행 계좌/투자 계좌 그룹, 기관/유형/계좌번호/기본 계좌 상태 |
| 결제수단 | 이름/유형/카드사/끝자리/연결 계좌/기본 여부 |
| 가구 구성원 | 구성원과 대기 중 초대를 분리한 사람 중심 리스트 |
| 월별 상세 | 월, 수입, 지출, 저축률을 한 묶음으로 보여주는 timeline 또는 metric list |

### 6.4 분석 드릴다운

분석 화면의 드릴다운은 요약 값을 다시 설명하지 않고, 사용자가 클릭한 숫자나 건수를 구성하는 원본 항목을 보여줍니다.

원칙:
- 차트 조각보다 텍스트와 숫자가 함께 있는 row를 우선 클릭 대상으로 삼습니다.
- 도넛 조각은 같은 기능을 row가 제공하면 1차 클릭 대상에서 제외합니다.
- Drawer 상단에 총액, 비율, 수익률 같은 요약 카드를 반복하지 않습니다.
- Drawer 본문은 원본 항목 목록을 중심으로 구성합니다.
- Drawer는 기본 20건과 전체 건수를 보여줍니다.
- 20건을 초과하면 `전체 기록 보기`로 기존 목록 화면 이동을 제공합니다.
- 수정/삭제는 분석 drawer 안에서 제공하지 않습니다.
- 월 단위로 데이터가 많아질 수 있는 항목은 drawer 대신 이동 확인 dialog를 사용합니다.

1차 드릴다운 UI:

| 화면 | 클릭 대상 | 패턴 |
|------|----------|------|
| 카테고리 분석 | 상세 내역 row | 가계부 기록 drawer |
| 결제수단 분석 | 상세 내역 row | 가계부 기록 drawer |
| 일별 지출 현황 | 일별 지출 막대 | 가계부 기록 drawer |
| 월별 수입/지출 | 수입/지출 막대, 월별 상세 수입/지출 영역 | 이동 확인 dialog |
| 주식 종목별 비중 | 종목 row | 주식 상세 drawer |
| 주식 계좌별 비중 | 계좌 row | 주식 상세 drawer |
| 종목별 계좌 분포 | 종목+계좌 row | 주식 상세 drawer |
| 수익/손실 TOP | 종목 row | 주식 상세 drawer |

## 7. 주요 화면 가이드

### 7.1 홈 (`/home`)

**역할**: 로그인 후 첫 화면, 자산 현황 요약 및 빠른 액션 제공

```
구성:
1. 인사말 + 총 자산 (가장 크게)
2. 빠른 액션 버튼 (자산 기록, 분석, 설정)
3. 자산 유형별 현황 진입점 (주식, 현금, 부동산, 기타)
4. 환율 정보 (작게)

원칙:
- 총 자산이 가장 눈에 띄어야 함
- 자산 유형 진입점은 클릭 시 해당 유형 페이지로 이동
- "준비 중" 유형은 비활성화 상태로 표시
```

### 7.2 자산 메인 (`/assets`)

**역할**: 자산 유형 선택 허브, 전체 자산 요약

```
구성:
1. 총 자산 요약
2. 자산 유형별 진입점 (클릭 시 해당 유형으로 이동)
   - 주식/ETF: 보유 종목 수, 수익률, [보유 현황] [거래 내역] 바로가기
   - 현금/예적금: 준비 중
   - 부동산: 준비 중
   - 기타: 준비 중
3. [+ 자산 기록] 플로팅 버튼

원칙:
- 자산 유형을 먼저 선택하는 계층 구조
- 각 유형별로 요약 정보 제공
- 확장 시 새 유형 진입점 추가만으로 대응
```

### 7.3 주식 분석 허브 (`/assets/stock/analysis`)

**역할**: 주식 분석 관점 선택

```
구성:
1. 종합 분석 진입점
2. 구성 분석 진입점
   - 소유자별
   - 위험도별

원칙:
- 허브는 데이터 조회 없이 분석 관점 선택만 담당
- 숫자와 차트는 각 분석 페이지에서 표시
- 전체 자산 분석 허브(`/assets/analysis`)는 제품 표면으로 두지 않음
```

### 7.3.1 주식 종합 분석 (`/assets/stock/analysis/overview`)

**역할**: 특정 자산 유형에 대한 상세 분석

```
구성:
1. 해당 자산 총액 / 수익률
2. 종목별 비중 차트 (도넛)
3. 종목별 상세 목록
4. 환율 정보 (해외 자산 시)

원칙:
- 해당 자산 유형의 전체 비중, 종목별 상세 제공
- 목록에서 종목, 계좌, 종목+계좌 row 클릭 시 분석 맥락 안에서 drawer를 열어 원본 보유 row와 최근 거래를 보여줌
- 차트 조각 클릭은 1차 상호작용에서 제외하고 row 클릭으로 커버
```

### 7.3.2 주식 구성 분석 (`/assets/stock/analysis/by-owner`, `/assets/stock/analysis/by-risk`)

**역할**: 주식 보유 현황을 소유자와 위험도 기준으로 분석

```
구성:
1. 소유자별/위험도별 분석 페이지
2. 선택된 기준 비중 차트 (도넛)
3. 기준별 상세 내역 목록

원칙:
- 각 분석 관점은 child route로 분리
- `by-asset-type`/자산군별 화면은 유지하지 않음
- 같은 화면 문법으로 차트와 목록을 연결
```

### 7.4 주식 보유 현황 (`/assets/stock/holdings`)

**역할**: 주식/ETF 보유 종목 목록

```
구성:
1. 상단 탭: [보유 현황] [거래 내역]
2. 필터/정렬 옵션
3. 종목 리스트
4. [+ 거래 추가] 버튼

원칙:
- 사용자-facing 테이블보다 컬렉션 UI를 기본으로 함
- PC에서는 넓은 리스트 또는 선택-상세 구조를 우선 검토
- 정렬/필터는 상단에
- 종목명, 투자금, 수량, 평균 매수가가 핵심 정보
```

### 7.4.1 주식 거래 내역 (`/assets/stock/transactions`)

**역할**: 주식/ETF 매수·매도 기록을 날짜 흐름으로 조회

```
구성:
1. 월 이동 + 필터 + [+ 거래 추가]
2. 월간 캘린더 또는 날짜별 요약
3. 선택 날짜 거래 목록
4. 거래 상세/수정/삭제 흐름

원칙:
- 날짜가 핵심 축이므로 단순 row 목록보다 캘린더/날짜별 목록을 우선함
- PC는 캘린더/날짜 요약과 선택 날짜 목록을 함께 보여줄 수 있음
- 모바일은 캘린더 -> 날짜별 목록 -> 상세/수정 흐름으로 분리함
- 거래 item은 종목명, 매수/매도, 금액을 우선하고 수량/단가/소유자는 보조 정보로 둠
- API는 우선 기존 거래 목록 데이터를 사용하고, 데이터 양 문제가 확인될 때 월 단위 서버 필터를 검토함
```

### 7.5 주식 거래 등록 (`/assets/stock/transactions/new/full`)

**역할**: 매수/매도 기록 입력

```
구성:
1. 거래 유형 선택 (매수/매도)
2. 종목 검색
3. 수량, 단가, 거래일 입력
4. 메모 (선택)
5. 저장 버튼

원칙:
- 한 화면에 모든 필드 (스크롤 최소화)
- 필수 입력만 먼저, 선택 입력은 접힘
- 저장 버튼은 항상 보이게 (sticky)
- 실시간 유효성 검사
```
