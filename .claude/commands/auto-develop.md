# 자동 개발 워크플로우

> 이 커맨드는 자동화 환경에서 실행됩니다. **사용자 상호작용 없이** 모든 Phase를 완료합니다.

## 실행 규칙

1. **절대 사용자에게 질문하지 않습니다.** 모든 결정은 자체적으로 판단합니다.
2. **시간 추적**: Phase 0에서 `date +%s`로 시작 시간을 기록합니다.
3. **시간 체크**: 각 Phase 시작 전 + 팀메이트 대기 중 **15분마다** `date +%s`로 경과 시간을 확인합니다.
4. **타임아웃**: 1시간 30분(5400초) 초과 시 현재 Phase를 마무리하고 Phase 4(정리)로 이동합니다.
5. **에러 처리**: 3번 재시도 후 실패하면 현재 진행상황을 커밋/푸시하고 종료합니다.
6. **커밋 규칙**: 각 Phase 완료 시 변경사항이 있으면 커밋합니다. Conventional Commits 형식을 따릅니다.
7. **품질 검증은 Developer가 담당합니다.** 리더는 품질 검증을 별도로 수행하지 않습니다.

---

## Phase 0: 초기화

### 0-1. 시작 시간 기록

```bash
date +%s
```

이 값을 **START_TIME**으로 기억합니다.

### 0-2. 브랜치에서 이슈 번호 추출

```bash
git branch --show-current
```

브랜치명 패턴에서 이슈 번호를 추출합니다:
- `feature/42-add-login` → `42`
- `fix/123-bug-fix` → `123`
- `/` 뒤의 첫 번째 숫자가 이슈 번호입니다

이슈 번호를 추출할 수 없으면 에러 메시지를 출력하고 종료합니다.

### 0-3. 이슈 조회

```bash
gh issue view {이슈번호} --json title,body,labels,milestone,assignees
```

조회 실패 시 종료합니다.

### 0-4. 부모 이슈 확인

이슈 본문에서 부모 이슈 참조를 찾습니다 (예: `Parent: #123`, `Part of #123`, `Tracking issue: #123`).
부모 이슈가 있으면 추가 조회합니다:

```bash
gh issue view {부모이슈번호} --json title,body,labels
```

### 0-5. 기존 PR 확인 및 진행 상황 파악

해당 브랜치의 열린 PR이 있는지 확인합니다:

```bash
gh pr list --head $(git branch --show-current) --state open --json number,url,title,body
```

#### Case A: PR이 이미 존재하는 경우 (이어하기)

기존 PR 번호를 **PR_NUMBER**로 기억합니다.

진행 상황을 파악합니다:
1. `git log origin/main..HEAD --oneline`로 기존 커밋 확인
2. PR 본문과 코멘트에서 이전 작업 진행 상황 확인
3. 변경된 파일 목록 확인: `git diff origin/main --name-only`

파악한 진행 상황을 기반으로 적절한 Phase부터 시작합니다:
- 커밋이 없거나 계획만 있는 경우 → Phase 1 (기획부터)
- 테스트/구현 코드가 있지만 미완성 → Phase 1 (기획 건너뛰고 개발 태스크만)
- 구현이 완료되었지만 품질 미통과 → Phase 1 (Developer에게 수정만 요청)

#### Case B: PR이 없는 경우 (첫 작업)

드래프트 PR을 생성합니다:

```bash
git push -u origin $(git branch --show-current)
gh pr create --draft \
  --title "[WIP] #{이슈번호} {이슈제목}" \
  --body "$(cat <<'EOF'
Closes #{이슈번호}

🤖 자동 개발 진행 중...

---
Generated by auto-develop
EOF
)"
```

PR 번호를 **PR_NUMBER**로 기억합니다.

## Phase 1: 팀 구성 및 기획

### 시간 체크

```bash
date +%s
```

`(현재 - START_TIME) > 5400` 이면 Phase 4로 이동합니다.

### 1-1. Agent Team 생성

다음과 같이 Agent Team을 구성합니다:

- **팀 이름**: `auto-dev-{이슈번호}`
- **Planner 팀메이트**: `planner` 에이전트 타입 사용, Sonnet 모델
- **Developer 팀메이트**: `developer` 에이전트 타입 사용, Opus 모델, **plan approval 필수**

Planner 스폰 시 다음 컨텍스트를 프롬프트에 포함합니다:

```
이슈 #{이슈번호}: {이슈제목}

{이슈 본문 전체}

라벨: {라벨 목록}

{부모 이슈 정보 (있는 경우)}

위 이슈를 분석하고 구현 계획서를 작성해주세요.
프로젝트 문서(.claude/docs/)와 코드베이스를 탐색하여 기존 패턴에 맞는 계획을 수립하세요.
계획서 작성이 완료되면 태스크를 완료 처리하세요.
완료 후에도 Developer의 질문에 답변할 수 있도록 대기해주세요.
```

Developer 스폰 시 다음 컨텍스트를 프롬프트에 포함합니다:

```
이슈 #{이슈번호}: {이슈제목}

Planner가 기획 태스크를 완료하면, 그 결과를 확인하고 TDD 방식으로 구현하세요.
애매한 부분이 있으면 Planner에게 직접 메시지를 보내서 확인하세요.
구현 완료 후 pnpm vitest run, pnpm type-check, pnpm biome check . --write가 모두 통과하는지 확인한 뒤 태스크를 완료해주세요.
품질 검증은 당신이 최종 담당이므로 반드시 통과시킨 후 완료하세요.
```

이어하기(Case A)인 경우, 이전 진행 상황도 Developer 프롬프트에 포함합니다:

```
[이전 작업 진행 상황]
{기존 커밋 내역}
{변경된 파일 목록}

위 진행 상황을 확인하고, 남은 작업을 이어서 진행하세요.
```

### 1-2. 태스크 생성

| 태스크 | 할당 | 의존성 |
|--------|------|--------|
| `기획: #{이슈번호} 구현 계획 수립` | Planner | 없음 |
| `개발: #{이슈번호} TDD 구현` | Developer | 기획 태스크 완료 후 |

이어하기(Case A)에서 기획이 이미 완료된 경우, 기획 태스크는 생략하고 개발 태스크만 생성합니다.

### 1-3. Planner 작업 대기

Planner가 기획 태스크를 완료할 때까지 대기합니다.
**대기 중 15분마다 시간을 체크합니다.**

## Phase 2: 기획 검토 및 커밋

### 시간 체크

`(현재 - START_TIME) > 5400` 이면 Phase 4로 이동합니다.

### 2-1. Planner 출력물 검토

Planner의 구현 계획서를 확인합니다:
- 요구사항 분석이 이슈와 일치하는지
- 테스트 계획이 포함되어 있는지
- 구현 단계가 구체적인지

부족하면 Planner에게 보완을 요청합니다.

### 2-2. 커밋

변경된 파일이 있으면 커밋합니다:

```bash
git add -A
git diff --cached --quiet || git commit -m "$(cat <<'EOF'
docs: #{이슈번호} 구현 계획

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"
git push
```

## Phase 3: 개발 감독

### 시간 체크

`(현재 - START_TIME) > 5400` 이면 Phase 4로 이동합니다.

### 3-1. Developer Plan Approval

Developer가 plan approval을 제출하면:

1. Developer의 구현 계획을 Planner의 계획서와 대조합니다
2. **일치하면**: 승인합니다
3. **불일치하면**: 구체적인 피드백과 함께 반려합니다 (예: "Planner는 X 패턴을 사용하라고 했는데, Y로 변경한 이유를 설명해주세요")

### 3-2. 개발 진행 모니터링

Developer가 TDD 개발을 진행합니다:
- Red → Green → Refactor 사이클
- Developer ↔ Planner 직접 소통 (필요 시)
- 품질 검증(vitest, type-check, lint)은 Developer가 수행

Developer가 개발 태스크를 완료할 때까지 대기합니다.
**대기 중 15분마다 시간을 체크합니다.**

### 3-3. 개발 결과 커밋

```bash
git add -A
git diff --cached --quiet || git commit -m "$(cat <<'EOF'
feat: #{이슈번호} {기능 설명}

- TDD 기반 구현
- 유닛 테스트 포함

Closes #{이슈번호}

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"
git push
```

## Phase 4: 마무리

### 4-1. 미커밋 변경사항 처리

```bash
git add -A
git diff --cached --quiet || git commit -m "$(cat <<'EOF'
chore: #{이슈번호} 중간 저장

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"
git push
```

### 4-2. 팀 정리

1. Planner에게 셧다운을 요청합니다
2. Developer에게 셧다운을 요청합니다
3. 모든 팀메이트가 종료되면 Team cleanup을 실행합니다

### 4-3. PR 업데이트

#### 정상 완료 시

```bash
gh pr edit {PR_NUMBER} \
  --title "#{이슈번호} {이슈제목}" \
  --body "$(cat <<'EOF'
## Summary
{구현 내용 요약 - bullet points}

## Test plan
{테스트 내용 요약}

Closes #{이슈번호}

🤖 Generated by auto-develop

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"

gh pr ready {PR_NUMBER}
```

#### 타임아웃으로 중단 시

PR을 draft 상태로 유지하고 코멘트를 남깁니다:

```bash
gh pr comment {PR_NUMBER} --body "$(cat <<'EOF'
## ⏰ 자동 개발 타임아웃

1시간 30분 제한으로 작업이 중단되었습니다.

### 완료된 작업
{완료된 Phase 목록}

### 남은 작업
{미완료 Phase 및 구체적 내용}

### 재개 방법
동일 브랜치의 워크트리에서 `/auto-develop`을 다시 실행하면 이어서 작업합니다.
EOF
)"
```
